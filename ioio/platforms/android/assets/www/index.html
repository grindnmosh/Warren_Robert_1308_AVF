  • Keeping in mind that you said Twitter would appear still in some lessons and that we should replace it in all cases with Instagram, my writing covers Instagram in place of Twitter where applicable.

  • Instagram could be used to as a built in login to the apps private areas or cross device/web storage for auto population of Data already entered onto a new device. This would also allow for instant syncing across the devices allowing Instagram to be used as cloud storage. I have never actually seen this in any app, used with Instagram. This is more commonly seen with Facebook or Google+ logins and occasionally Twitter logins. I prefer the apps I have seen that give you the choice between direct login with no cloud support or the multiple social network login options with cloud support. I like the apps that have a simple backup sync with more popular direct cloud options like Dropbox or Box. I intend to use the Facebook Like API, the Google+ +1 API, the Twitter Follow API, Evernote syncing API as well as include the Dropbox API for backing up Data. I just need to learn how to implement these API features. I also want to make the login options of some or all of these available. I would love to say this would all happen this month but we will see.

  • JSON-P is an interesting concept, in the fact that is lets use Data from another domain. This allows you to pull data from the API’s source websites with no issues. Cross platform the browser generally denies data unless wrapped in script tags, which makes the data useless beyond evaluating it. JSON-P uses this loophole to gain access to cross-domain data and wraps it in a way that makes it fully usable. Without JSON-P a call for data would involve several steps of calling your server, which calls on the API’s server, which in turn sends it back to your server then to you. That is a lot slower in getting returned data than directly getting it and only JSON-P can currently do this effectively. So JSON-P apps definitely provide an advantage in getting data returned to you faster.

  • Some of the APIs that blend best with native functions that I found actually blend between native features. The camera would blend well with any of your social media services. Storage would blend well with cloud services like Dropbox, Box or even Facebook or Google+. The Network feature works well with the in-app web browser, the social networks, the cloud storage, the sharing features using email or messaging and anything else that needs to pull information from or even sending information to the web. Contacts can be used to sync with any of the incorporated services or to populate into services. These would sync well with sharing options using social network APIs. The options are limitless and these are only some general examples.

  • My intentions this month is to begin my social network promotional integration and multi device sync support into my app and add the features needed to take full advantage of native features like contacts, calendar, network, internal storage and when applicable SD card support. I know SD storage will only be an option for the Android side for most devices but this is a highly request feature of Android users so I do intend to implement it. I intend to have at least the basic functionality to make the app fully useable by the end of this month and have it available on both app stores. This is an app that I already enjoy using in the mobile web version I have and I have had requests to be a beta tester for this app when it becomes available for such. I already discussed in the first paragraph some detailed desires for the use of API’s in this app.

  • We have learned so much up till now and honestly it is really jumbled up in my head as to where or when I learned what. It is even jumbled which techniques I have just formed in combination of what I have learned in class, from others or just trial and error. But I will try to break down these three categories. As far as research, this is the most important step in any project. You should a lot time before beginning the structuring of the project itself to research the why, how to and effective use of the big picture of what you want to do. This in turn creates a need for additional research on the components that need to be incorporated, some of which you had already planned and some that you hadn’t previously thought of. This creates it’s own research. Once you have completed your thorough research you are ready to begin the design stage. The design stage should include two main pieces. They are our wireframe and your flowchart. As you build these you find the need for even more components to complete your goals returning you to the research stage. This is also the stage to plan your apps branding, which may change as you go. Once your planning is complete, it is time to begin the design stages. This is the point when you start writing your primary code, which has been html5 so far. We then make it look nice with CSS. We finally make it function with our JavaScript or JavaScript libraries such as jQuery. The final step is to add data sources and native functionality to make the app function as we envisioned it. One of the biggest things I have learned and something I want to ensure is included, is to not over think it. As humans that is our biggest flaw.
